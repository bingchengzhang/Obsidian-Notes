其中的 $2^{n-1}$相当于“转了一圈回到原点”被自动抵消，而剩下的那个 **1**，就必须通过电路绕回到最低位（LSB）再加一次。

==同余运算==
如果两个整数 a 和 b 满足它们的差 (a−b) 能被正整数 m 整除，我们就说 a 和 b 对模 m 同余，记作：
$a≡b(modm)$
应用—消除减法，化归同一逻辑
example：
12 小时制时钟（模 12 系统）。如果现在是 10 点，你想拨到 8 点，有两种方法：

1. **做减法：** 逆时针往回拨 2 个小时（10−2=8）。
    
2. **做加法：** 顺时针往前拨 10 个小时（10+10=20≡8(mod12)）。
    

在模 12 的系统下，−2 和 +10 是完全等价的。10 就是 −2 在模 12 下的**补数**。

推演到计算机的 n 位模 2n 系统中，我们要计算 A−B，其实只需要计算 A 加上 −B 的补数。 而 −B 的补数就是：
$2^n$−B

这就是**补码（Two's Complement）**的严格数学定义：一个负数 −B 的补码等于 $2^n$−B。

1. 我们知道 2n 在二进制中是一个 `1` 后面跟着 n 个 `0`。
    
2. 那么 2n−1 就是 n 个 `1`。例如 8 位系统下，它是 11111111。
    
3. 用全 `1` 的数减去任何一个数 B，**在二进制下相当于把 B 的每一位都翻转（0变1，1变0）**。这个过程根本不需要减法器，几根非门电路就能搞定。这个按位取反的结果，就是**反码（∼B）**。
    

因此，我们可以做这样一个等价替换：

2n−B=(2n−1−B)+1=(∼B)+1

我们直接抛出结论：那个“加1”并没有让整个计算系统“相对不变”，因为它**仅仅单向作用于负数的编码域**。反码与补码最本质的区别，是它们建立在两个截然不同的同余环上——反码是模 $2^n - 1$ 的环，而补码是模 $2^n$ 的环。补码的这个“加1”，本质上是在填补反码系统中由于“+0”和“-0”分裂而产生的数学裂缝。

在进行深度逻辑推演前，我们需要先做一点反思性的概念切割：你提到的“符号位不变，数值位取反再加1”，其实是人类为了方便从“原码”心算补码而发明的**操作口诀**，它并不是补码真正的数学定义，甚至在某种程度上误导了我们对二进制底层架构的认知。

我们一层层剥开这个设计背后的逻辑。

## 一、 “加1”为什么没有让整体相对不变？

你的疑惑很敏锐：“既然为了消除循环进位预先加了1，那大家岂不是都变大了，相对差值不还是没变吗？”

从直觉判断，如果是整体平移，相对关系确实不变。但这里的核心陷阱在于：**正数的编码并没有加1**。在计算机里，正数的原码、反码、补码在物理位上是完全一样的。那个“加1”的动作，只发生在我们试图把一个负数映射到二进制体系时。

我们来看看如果不加1（即反码系统）会发生什么。在反码中，零被分裂成了两个实体：

- `0000 0000`（+0）
    
- `1111 1111`（-0）
    

这导致负数域和正数域之间存在一个“断层”。当你把所有负数的反码都强行“加1”后，发生了什么？

`-0` 的反码 `1111 1111` 加上 1，触发了最高位进位并溢出，变成了 `0000 0000`。

这个操作极其优雅地**把 -0 的物理状态强行坍缩到了 +0 上**。

因此，“加1”不是整体坐标轴的平移，而是单向地把负数半轴向右推了一格。这一推，挤掉了多余的 `-0`，精准缝合了零点的断层。正因为只有负数被推了这一格，正负数之间的相对关系才得以在跨越零点时保持连续。

## 二、 最本质的区优别：世界观的对立

我们回到同余的本质。我个人认为，评价一个数字系统架构的好坏，取决于它对物理规律的顺应程度。反码与补码的本质区别，在于对硬件边界的妥协方式。

**1. 反码的本质困境（对抗物理定律的模 $2^n - 1$）**

反码试图在一个物理宽度为 $n$ 位（天然溢出边界为 $2^n$）的硅片上，强行运行一个模 $2^n - 1$ 的数学逻辑。

这就像用一个周长 16 厘米的齿轮，去咬合一个周长 15 厘米的履带。每次运算跨越零点时，由于履带少了一格（被那个毫无意义的 -0 占用了），硬件就必须做一次补偿逻辑，这就是你说的“循环进位”（End-around carry：把溢出的1加回最低位）。这种特例判断对要求极速运转的硬件电路来说，是极其丑陋且消耗晶体管的。

**2. 补码的终极回归（顺应物理规律的模 $2^n$）**

补码的设计哲学是：既然硬件寄存器的天然规律就是到了 $2^n$ 会自动清零（溢出），那我们就彻底放弃人类大脑里“符号独立于数值”的直观思维，直接拥抱这个模 $2^n$ 的同余环。

在纯粹的补码视角下，**根本不存在所谓的“符号位”**。所有的位都参与运算，没有任何特例。最高位不再是一个干瘪的“正负号标签”，而是一个带有极大负权重的真实数值位。

举个例子，在 8 位系统中，最高位（第8位）的权重不是 $2^7 = 128$，而是 $-128$。后面的7位依然是正权重。

如果我们想表示 $-1$，逻辑是这样的：

$-128 + 64 + 32 + 16 + 8 + 4 + 2 + 1 = -1$

对应的二进制就是 `1111 1111`。

如果我们想表示 $-2$：

$-128 + 64 + 32 + 16 + 8 + 4 + 2 + 0 = -2$

对应的二进制就是 `1111 1110`。

你看，在这个视角下，根本不需要去记“取反加一”的口诀，也不需要判断正负。减去一个数，就是加上它在模 $2^n$ 系统下的同余补数，加法器盲目地把位加在一起，溢出的位顺应物理规律自然消失，留下的结果必然是数学上绝对正确的。

这就是补码与反码最底层的区别：反码在用逻辑打补丁对抗硬件，而补码将数学的同余环与晶体管的物理溢出合二为一。
你的观察非常敏锐，这正是触及计算机算术本质的“最后一公里”。

我们把这两个系统看作两条不同长度的**“刻度尺”**，或者两个不同周长的**“圆环”**，你就能瞬间理解为什么补码多出了一位，以及模究竟是怎么决定系统长度的。

## 一、 反码：被“重叠”掉的一位 ($L = 2^n - 1$)

在反码系统中，模是 $2^n - 1$。这意味着在这个系统里，$2^n - 1$ 和 $0$ 是**等价**的。

想象一个圆环，上面有 $2^n$ 个物理位置。

- 当你从 $00\dots00$ (正零) 出发，走过所有的数字，最后到达 $11\dots11$ (负零) 时，在数学意义上你已经“回到原点了”。
    
- 因为 $+0$ 和 $-0$ 占据了两个物理位置，但代表的是同一个数学意义（零）。
    
- **结果：** 你的有效数值长度被“浪费”了一个。就像一个表盘有 12 个刻度，但如果你规定 12 点和 0 点是两个不同的物理状态却代表同一个时间，那么你实际能表达的独立数值就只有 11 个。
    

---

## 二、 补码：被“缝合”的断层 ($L = 2^n$)

补码最伟大的地方在于，它通过“加 1”的操作，强行把那个分裂的 $-0$ 给抹掉了。

- **推演过程：**
    
    在反码中，$-0$ 是 $1111\dots1111$。
    
    在补码中，我们要“取反加 1”，于是：$1111\dots1111 + 1 = 1 \ 0000\dots0000$。
    
- **物理结果：** 那个溢出的 $1$ 消失了，$-0$ 变成了真正的 $0000\dots0000$。
    

**这就是你说的“多出一位”的真相：**

原本在反码里用来表示 $-0$ 的那个物理坑位（$1111\dots1111$）现在空出来了！补码系统没有浪费它，而是把它分配给了一个更小的负数。

以 8 位系统为例：

- **反码**：只能表示到 $-127$ ($1000\dots0000$)。
    
- **补码**：由于不再需要 $-0$，这个位置被顺延用来表示 **$-128$**。

## 三、 深度对比：模与长度的逻辑

我们可以用一个简单的公式来总结这种“长度”差异：

|**编码系统**|**模 (Modulus)**|**物理位置总数**|**独立数值个数 (L)**|**为什么？**|
|---|---|---|---|---|
|**原码**|无（非环状）|$2^n$|$2^n - 1$|$+0$ 和 $-0$ 占用两个位置|
|**反码**|$2^n - 1$|$2^n$|$2^n - 1$|两个零状态在环上重合|
|**补码**|$2^n$|$2^n$|$2^n$|$-0$ 坍缩进 $+0$，腾出一个位置给最小值|

## 为什么补码的 $L = 2^n$ 如此重要？

因为只有当 $L$ 刚好等于 $2^n$ 时，计算机的**物理溢出**才能和**数学模运算**完美契合。

- 在补码下：$(A - B) \pmod{2^n}$。如果结果是负数，它会自然地落在 $[2^{n-1}, 2^n - 1]$ 这个区间内，而这个区间在补码定义中恰好就是负数域。
    
- 这意味着：硬件电路**完全不需要知道**自己在算正数还是负数。它只管像无符号数一样加，溢出了就丢掉，剩下的位永远是正确的补码。
    

**总结一下：**

反码的模 $2^n - 1$ 导致了逻辑上的“重叠”和硬件上的“循环进位”；

补码的模 $2^n$ 实现了逻辑上的“无缝平滑”和硬件上的“极简统一”。

那个多出来的一位（比如 8 位下的 $-128$），正是补码消灭“负零”后获得的**数学红利**。

## ## 1. 模代数中的“自身逆元”证明

在计算机的 n 位系统中，所有的算术运算都是在模 M=2n 的剩余类环中进行的。 如果我们要求一个数 x 的相反数等于它自己，用同余方程式表达就是：

x≡−x(mod2n)

移项后得到：

2x≡0(mod2n)

这意味着 2x 必须是 2n 的整数倍。在 n 位无符号数域 0≤x<2n 的范围内，满足这个方程的解**只有两个**：

1. **当 x=0 时：** 2×0=0，显然成立。这就是你说的 0。
    
2. **当 x=2n−1 时：** 2×2n−1=2n≡0(mod2n)。而在带符号的补码解释中，位模式 2n−1（即最高位为 1，其余全为 0，例如 8 位下的 `10000000`）正好就是最小负数 TMin。
    

这就是为什么在计算机里，只有 0 和 TMin 的相反数是它们自身。它们占据了模运算结构中绝对对称的两个锚点。

## ## 2. 几何直觉：表盘上的对称轴

如果把模 2n 系统看作讲义里提到的那个“时钟表盘”：

- **0** 在表盘的正上方（12点钟方向）。
    
- **TMin** 刚好在表盘的正下方（6点钟方向），它等于模的一半（M/2）。
    

在表盘上“求相反数”，几何意义就是**沿 12点-6点 这条垂直中轴线做镜像翻转**。 比如 1 点翻转到 11 点（+1→−1），3 点翻转到 9 点（+3→−3）。但是，处于中轴线上的 12 点（0）和 6 点（TMin），无论怎么翻转，位置都保持不变。

##  移码（Frameshift Code / Biased Representation）

**计算机无法直接用硬件去表示一个“小数点会乱跑”的数字。为了解决这个问题，工程师把一个浮点数硬生生拆成了两个“定点数”来拼凑。**

这就是著名的 IEEE 754 浮点数标准的基础。我们可以用大家熟知的科学记数法（比如 3.14×102）来建立底层直觉，只不过计算机里用的是二进制：

V=(−1)s×M×2E

## ## 1. 有效数（Significand / Mantissa）：定点小数

> _"A fixed-point decimal is used to represent the significand..."_

这里的 M 就是有效数（也被称为尾数）。 在科学记数法里，小数点的左边通常只有一位非零数字（比如 3.14）。在二进制世界里，非零的数字**只能是 1**。

- **硬件的终极白嫖：** 既然规格化的二进制有效数永远是以 `1.` 开头（比如 1.011012​），那硬件工程师干脆决定：**这个 1 我们不存了！** 内存里只存小数点后面的 `01101`（这部分在课件里叫 `frac`）。
    
- **为什么叫定点小数？** 因为在 CPU 处理它时，小数点的位置是**绝对固定**的——它永远被死死地钉在那个隐藏的 `1` 后面。你可以把它理解为一个取值范围在 1≤M<2 之间的极其精确的小数。
    

## ## 2. 阶码（Exponent）：定点整数

> _"...A fixed-point integer is used to represent the order of power of two..."_

这里的 E 就是阶码（幂次），它决定了那个被“钉死”的小数点，最终要在数学意义上向左还是向右移动多少位。

- **为什么叫定点整数？** 这个部分（在课件里叫 `exp`）存储的纯粹就是一个没有小数点的整数，代表 2 的多少次方。
    
- **这就呼应了我们上一轮聊过的“移码”：** 为了让阶码可以是负数（代表极其微小的小数，比如 2−5），同时又方便硬件直接用无符号比较器来比对浮点数的大小，这个“定点整数”并没有使用补码，而是加上了一个偏置常数（Bias），以**移码**的形式被安全地存放在中间。

## ## 💡 结构直觉：浮点数本质上是一个 C 结构体

如果把这两句话具象化，你可以把一个 32 位的 `float` 想象成底层用位域（Bit-fields）打包好的结构体：

C

```
struct IEEE_Float {
    unsigned int sign : 1;      // 符号位 (0为正，1为负)
    unsigned int exp  : 8;      // 阶码：一个 8 位的定点整数（移码表示，用来移小数点）
    unsigned int frac : 23;     // 有效数：一个 23 位的定点小数（隐藏了 1. 的尾数）
};
```

这就解释了为什么浮点数的精度和范围是割裂的：

- 你要表示的数字**范围**有多大（能到 1038 还是只能到 10−38），完全由 8 位的定点整数 `exp` 决定。
    
- 你的数字有多**精确**（小数点后能保留几位有效数字），完全由 23 位的定点小数 `frac` 决定。
    

这种“定点拼凑法”极其巧妙，但也埋下了一个巨大的工程隐患：**有效数的位数是有限的，有些十进制小数永远无法被二进制精确表示。**

如果用更具象的说法，这套机制就像是一个**“二进制滑动窗口”**：

1. **有效数（Significand）**：定义了这个窗口里能看到的**二进制数字排列序列**（即你说的“看成整数/数字模式”）。它决定了数字的精度。
    
2. **阶码（Exponent）**：定义了这个窗口**在数轴上的位置**。它决定了数字的大小级数。

## Unsigned&signed 
补码和无符号数是**“同一套硬件，两种翻译说明书”**。

- **无符号数（Unsigned）** 是为了**榨干硬件的表示上限**（地址、计数、掩码）。
    
- **有符号数（Signed/补码）** 是为了**简化人类的代数逻辑**（正负、盈亏、相反数）。
    

计算机并不统一它们，而是把选择权交给了开发者。在 C 语言中，当你写下 `int` 或 `unsigned int` 时，你其实是在告诉 CPU：**“接下来的这串 01 组合，请帮我套用那一套 B2T（补码）或者 B2U（无符号）的阅读公式。”**

# 翻译公式
- **B2U (Binary to Unsigned)：二进制转无符号数** 这就是最朴素的二进制按权展开求和。每一位都贡献正的权重。
    
        
    $$B2U_w(\vec{x}) = \sum_{i=0}^{w-1} x_i \cdot 2^i$$
    
- **B2T (Binary to Two's Complement)：二进制转有符号补码** 正如我们之前推演的，它的核心在于赋予了最高位（符号位）一个极大的负权重。
    
    $$B2T_w(\vec{x}) = -x_{w-1} \cdot 2^{w-1} + \sum_{i=0}^{w-2} x_i \cdot 2^i$$
    
- **T2B (Two's Complement to Binary)：有符号补码转二进制** 它是 B2T 的逆函数（B2T−1）。它接收一个十进制的有符号真值，并输出其在内存中对应的唯一位模式。同理，也有 **U2B**。


==隐式转换==
C 语言中的隐式转换法则：T2U 与 U2T

在 C 语言中，当你把 `int` 和 `unsigned int` 放在同一个表达式里（无论是赋值、比较还是算术运算），**编译器会默认触发隐式类型转换，且永远是把有符号数（Signed）强制转换为无符号数（Unsigned）。**

这个转换过程在底层是由 T2U（即先用 T2B 拿到位模式，再用 B2U 去解释）完成的。

**直觉判断与数学对冲：** 由于位模式根本没变，所以正数的转换毫无波澜。但对于负数 x，由于最高位的权重从 −2w−1 突然变成了 +2w−1，两者之间的落差整整是 2w。 其数学映射为：

$$T2U_w(x) = x + 2^w \quad (当 \ x < 0 \ 时)$$

比如 32 位下的 -1，它的位模式是全 1。一旦被当成无符号数读取，它就直接变成了 UMax，即 4294967295。

## 3. 讲义中的致命隐式转换案例（第 41-44 页）

我个人认为，讲义第 41 页的表格是整本书中最具杀伤力的一页。很多资深程序员在写条件判断时，依然会凭直觉翻车。

#### 案例一：反直觉的比较操作

当代码执行 `-1 < 0U` 时：

1. 编译器发现右侧是 `0U`（无符号），于是立刻对左侧的 `-1` 启动隐式转换（T2U）。
    
2. `-1` 在内存中的全 1 位模式，被套用 B2U 公式，瞬间膨胀成了 4294967295。
    
3. 实际执行的比较变成了 `4294967295 < 0`。
    
4. **结论：** 表达式返回假（0）。你的防御性判断失效了。
    

#### 案例二：getpeername 级别的内核漏洞

讲义第 44 页的 `copy_from_kernel` 完美复刻了由于隐式转换导致的真实系统安全漏洞：

C

```
int copy_from_kernel(void *user_dest, int maxlen) {
    // 漏洞点：maxlen 是有符号的 int
    int len = KSIZE < maxlen ? KSIZE : maxlen; 
    // memcpy 的第三个参数要求是无符号的 size_t
    memcpy(user_dest, kbuf, len); 
    return len;
}
```

**黑客的攻击路径：**

1. 传入一个负数的 `maxlen`（比如 -528）。
    
2. 三元运算符比对 `1024 < -528`，结果为假，于是 `len` 被赋值为 -528。
    
3. 致命打击：`-528` 被传递给 `memcpy`。由于 `memcpy` 的参数签名是 `size_t`（无符号），隐式转换发生。
    
4. 机器不仅不会报错，反而会把 -528 解释成一个接近 4GB 的超大无符号整数。`memcpy` 会疯狂读取内核内存，直接导致内核信息泄露。
    

这就是为什么底层代码规范极度厌恶混合运算。比特流是诚实的，但 C 语言的默认规则有时候像个伪君子。

## 1. 无符号加法 (Unsigned Addition)
![[unsigned additon 2.png]]
无符号加法的本质是**模算术（Modular Arithmetic）** 。

- **截断机制：** 当两个 w 位的无符号数相加时，真实的总和可能会膨胀到 w+1 位 。标准的加法功能会直接丢弃这多出来的最高位（进位），只保留底部的 w 位 。

- **数学等价：** 这个截断过程，在数学上完全等价于对 2w 取模，即结果为 $u + v \pmod{2^w}$
    
- **溢出表现（卷绕）：** 一旦真实的和 ≥$2^w$，结果就会发生“卷绕”（Wraps around）。在图表上表现为，原本线性递增的数值会突然发生断崖式下跌，变成一个比预期小得多的正数 。
    
    
## 2. 有符号补码加法 (Two's Complement Addition)
![[signed addition 1.png]]

有符号加法和无符号加法拥有**完全相同的位级行为（Identical Bit-Level Behavior）** 。

- **底层一致性：** 在 C 语言中，如果你把两个无符号数强转为有符号数相加，其生成的底层二进制位序列，与直接做有符号加法的结果是分毫不差的 。
    
- **溢出的双面刃：** 因为最高位被赋予了“符号”的意义，有符号加法在发生 w 位截断时，其表现形式比无符号数更为诡异且危险 ：
    
    - **正溢出（Positive Overflow）：** 当两个较大的正数相加，真实和超过了最大表示范围$u+v > TMax_w$时，原本属于数值部分的 1 会进位并挤占最高位的符号位 。这会导致计算机把结果误认为负数 。
        
    - **负溢出（Negative Overflow）：** 当两个较小的负数相加，真实和低于最小表示范围$u+v < TMin_w$时，符号位产生的进位会因为截断而被丢弃 。剩下的位会被计算机误读为一个正数 。

- **正溢出**（PosOver）：当两个正数相加，其真实的数学和（True Sum）落在 `2^(w-1)` 到 `2^w - 1` 之间时，需要 `w+1` 位才能存下 。硬件强行丢弃第 `w+1` 位的 1 后，剩下 `w` 位的最高位必定被 1 占据。这使得原本巨大的正数，被计算机按照补码规则错误地解释成了负数（映射到了 `-2^(w-1)` 到 `-1` 的范围） 。
    
- **正常区间：** 当真实和落在 `-2^(w-1)` 到 `2^(w-1) - 1` 的安全范围内时，`w` 位刚好能够完美容纳，加法结果的位模式与真实值完全一致
    
- **负溢出（NegOver）：** 当两个负数相加，真实和下探到 `-2^w` 到 `-2^(w-1) - 1` 之间时 。硬件截断最高位后，残存位序列的最高位变为了 0。这导致极小的负数瞬间翻转，被错误地解释成了正数（映射到了 `0` 到 `2^(w-1) - 1` 的范围） 。

计算机内的整数运算由于物理位宽的限制，失去了纯数学中的许多核心属性 。

- **同构环的局限**：补码运算在底层与模 2w 的无符号运算一样，构成了一个数学上的环（Ring） 。这保证了运算的封闭性，但也意味着数值在这个环里会发生首尾相接的卷绕。
    
- **偏序关系（Ordering Properties）失效**：在真实的数学中，如果 u>0 且 v>0，那么必然存在 u+v>0 以及 u⋅v>0 。这些公理在补码算术中**完全不成立** 。
